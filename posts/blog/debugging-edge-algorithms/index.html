Debugging Edge Algorithms

Debugging Embedded C Python
USER: SYS_ADMIN | DATE: 2026-03-21

Simulation vs Bare Metal

While working on deploying an algorithm from simulation to actual device. It can be quite tricky to debug a black box.

This one time I deployed a sensor fusion algorithm. While, working on a POC simulation has its own hoops to jump over. Porting it to actual hardware is a whole another battle.

The amount of moving parts and things to keep track of increases exponentially in an actual implementation. Because you aren’t just putting a routine that can run at its own pace. It needs to run in a delicate balance of memory and time utilization allowing all the other threads running in the processor to not be hindered by it.

Hardware Bottleneck

The set-up I had was not very wide. All I could get was a channel dump .wav from the device, basically what the sensor was recording. Now, to know for sure the internal state-space of the algorithm. I had almost no way to see them unless I had other UART taps to dump those information as well.

The Hack: Hijacking the Audio Channel

Instead of asking for a different hardware revision or building a complex packet-parsing layer to separate logs from audio on the host machine, I decided to work within the constraints of the .wav format itself.

Instead of writing the audio packets for how the .wav file is written and read. I created a struct instance inside the SDK allowing me to decide the structure in which I will write for each sample’s packets.

Since, the hardware pipeline was setup for 16000Hz. I was able to create internal_state_packet instead of the data_packet for audio. Which was being written to the .wav file in the running machine. This came along with a paired python script which picks up the audio file and then decrypts or unpacks the file reversing the way in which the internal_state_packet was written.

Catching the Edge Case

Implementing this architecture allowed me to be able to dump any state or variable inside the algorithm and be able to see how it develops with time under different scenarios. Following this I was able to confirm the existence of a few bugs that I had suspected and alongside that I was also able to observe unexpected behavior in certain scenarios.

Narrowing down the testing cycles and giving more confident in the working and non-working cases of the algorithm. Especially being able to see the internal states development before it derails saved a lot of debugging and auditing cycles for me. Causing a quick delivery of the tested and working algorithm in a explained and transparent manner.

The Takeaways

Working close to the edge sometimes need you to have a different perspective. Compared to working in a walled garden of a cloud development No shade to that though but just having to work on edge always teaches you to think out of the box to solve a problem. As sometimes the solution to quickly unblock a problem is not to build new debugging toolset but to re-use already existing tools creatively.

By embedding state data directly into the audio pipeline, we avoided a hardware redesign, validated the algorithm’s behavior on the edge, and ultimately shipped a more resilient model.
